# HANDOFF — Lane B: Video proof-of-work + live watch mode

Branch: `claude/video-proof`

## What was built

1. **`src/project-video.ts`** (new) — flow recording built on the existing Playwright/screenshot infrastructure:
   - `recordProjectFlow(project, requestText, options)`: detects a running dev server via the existing `findProjectWebUrls`, derives up to 4 scored steps from the request text (`deriveFlowSteps`), records a Playwright video (1280x720, ~30s cap, ~1.5s dwell per step) while performing those steps (click via the existing `bestNavigationCandidate` scoring against visible clickable elements, or scroll via `page.mouse.wheel`), then reads back the `.webm`. If `ffmpeg` is on PATH it transcodes to `.mp4` for maximum inline-playback compatibility (verified working end-to-end with the ffmpeg on this machine); otherwise it attaches the `.webm` as-is.
   - Size-cap handling: `decideSizeCap` is a pure decision function (accept / shrink / fallback) against Discord's 8MB default attachment limit. On first oversize, one retry happens at a reduced viewport (960x540) and duration (18s). If still oversized, `recordProjectFlow` returns a `screenshot-fallback` outcome instead of a video.
   - `isUiRelatedTask(taskText, changedFiles)`: pure heuristic (UI vocabulary in the task text, or UI-flavored changed file extensions/directories) used to decide whether a completed `/do` task deserves proof capture.
   - `listChangedFiles(project)`: `git diff --name-only HEAD` + untracked files, feeding the heuristic above.
   - `selectRecentFrames` / `computeGifPageLayout` / `buildTimelapseGif`: pure frame-selection and sharp page-layout math, plus the actual animated-GIF composition (verified end-to-end with real sharp-generated frames in tests — produces a valid multi-page `GIF89a` buffer).
   - Exported previously-private helpers from `src/project-screenshot.ts` (`canReach`, `sanitizeFilePart`, `projectScreenshotOrigins`) for reuse instead of duplicating them.

2. **`/clip` command** (`src/commands.ts`, handled in `src/index.ts` near the existing `/snip` handler): `target` (required, natural language or path/URL), optional `project` (autocomplete, reuses the existing `project` autocomplete wiring — no changes needed there since it's keyed by option name), optional `steps` (free-text extra actions). Follows the exact same access pattern as `/snip`: `ensureProjectAccess`, then additionally gated by the existing `screenshotPolicyMessage` (deny/approval project policy), since recording is more sensitive than a single screenshot. Posts the recording (or, on fallback/unavailable, a screenshot or an explicit reason) with metadata: URL, viewport, steps actually performed, console errors seen.

3. **Auto proof on `/do`**: `/do` now runs through a new `executeDoInteraction` (kept separate from the shared `executeInteractionRequest` used by `/ask`, retries, etc. to avoid touching those paths). After a successful action-mode task, it checks `isUiRelatedTask(task text, git-changed files)`; if related, it calls `recordProjectFlow` and attaches the resulting video (or screenshot fallback, or an explicit "Proof capture unavailable: ..." note — never silent). Recording failures are caught and degrade to a note; they never fail the task itself.

4. **Live watch mode**: while `/do` runs, if `watch` (new boolean option, default `true`) is on and a dev server is detected for the project, a session captures a screenshot via the existing `captureProjectScreenshot` roughly every 20s and swaps it onto the running task message (same message the progress/task-control UI already lives on) without disturbing the progress text/components. All edits to that Discord message (progress-phase edits and watch-frame edits) are serialized through a small promise-chain queue to avoid interleaved/racing `editReply` calls. On completion, the last up-to-12 captured frames are stitched into an animated GIF (`buildTimelapseGif`, via `sharp`'s `{ raw, animated, pageHeight }` page-composition, confirmed working) and attached alongside the proof video/screenshot in the final message edit. Watch mode is skipped entirely (no capture, no note) when project policy blocks/gates screenshots, consistent with the deny-by-default access model.

5. Docs: added a `/clip` bullet and updated the `/do` bullet in `README.md`; added a "Video proof-of-work and live watch mode" bullet under Current Features in `docs/DEVBOT_PRODUCT_PLAN.md`.

## How to verify manually in Discord
1. Start a local frontend dev server for a configured project (Next/Vite/etc.) so `findProjectWebUrls` detects it.
2. `/clip target:"click settings, then scroll down" project:<name>` — expect a playable video attachment (webm or mp4 if `ffmpeg` is on PATH) with a metadata message (URL, viewport, steps performed, console errors).
3. `/do task:"change the button color on the settings page" project:<name> watch:true` — expect the task message to swap in fresh screenshots roughly every 20s while it runs (visible if the task runs long enough to hit a 20s tick), and on completion see either an attached recording, a screenshot-fallback note, or an explicit "Proof capture unavailable" note, plus an attached timelapse GIF if watch mode captured any frames.
4. `/do task:"..." watch:false` — confirm no periodic screenshot swaps happen.
5. Set a project's `.devbot/project.json` `screenshotPolicy` to `deny` and confirm `/clip` shows the policy-blocked approval card instead of recording, and that `/do` on that project produces no proof note at all changes (watch/proof are silently skipped, matching the deny-by-default policy).

I did not start the bot against Discord — verified via `npm test` (build + `node --test`) plus standalone Node scripts that exercise `recordProjectFlow` end-to-end against a real local HTTP server (confirmed a working `.mp4` with correctly derived steps, and a correct "no server detected" `unavailable` outcome) and `buildTimelapseGif` against real sharp-generated frames.

## Known limitations / risks
- Watch mode launches a fresh headless Chromium via `captureProjectScreenshot` on every ~20s tick; on a slow machine or a long-running task this is a non-trivial number of browser launches. Acceptable given the existing per-request screenshot pattern elsewhere in the codebase, but a future lane could add a longer-lived single browser/page for watch mode instead of relaunching.
- `deriveFlowSteps`'s phrase-splitting heuristic is intentionally simple (split on commas/semicolons/"then", filter to short-or-actionable phrases). It will sometimes pick fewer than 4 steps or none at all for oddly-phrased requests; `recordProjectFlow` still records a plain dwell-and-screenshot flow of the landing page in that case, and `stepsPerformed` in the reply metadata reports exactly what was attempted so this is never silently wrong.
- The final `/do` message edit intentionally omits `files` on progress-phase edits (routing/gathering-context/running-codex) so a previously-swapped watch screenshot isn't wiped out early; only actual watch ticks and the final completion edit change attachments.
- Size-cap fallback-to-screenshot path and the reduced-size retry path are covered by the `decideSizeCap` unit tests and manually reasoned through, but weren't exercised end-to-end with an actual >8MB recording (would require an artificially large capture to trigger in this environment).
- No changes were made to `mention.ts` / `executeMessageRequest` (the `@devbot do ...` mention path); auto-proof and watch mode are scoped to the `/do` slash command per the brief's explicit wording. If the wave wants parity for mention-triggered action tasks, that would be a follow-up.

## Review round 1 (maintainer requested changes)

Bernard requested changes; his review is the spec. Each blocking issue → fix + test below. New commits were appended on top of the reviewed history (no rewrite).

1. **Proof captured the wrong code (false confidence).** Action `/do` tasks always execute in an isolated worktree (`executionProject.root = worktree.path`), while the detected dev server serves the unchanged source checkout. So `captureCompletionProof` recorded the source server and presented it as proof of the isolated change (and `listChangedFiles(source)` was even empty because the diff lives in the worktree). Fix: `runProjectRequest` now reports `isolated`/`workspaceBranch` in `ProjectRequestResult`; `captureCompletionProof` takes that and, for isolated tasks, returns an honest note (`isolatedProofNote`) instead of recording — "proof capture skipped: this action ran in an isolated worktree … a recording would show the unchanged source app. Run `/clip` after merge." No recording is ever presented as proof of an unmerged change. Explicit `/clip` is unchanged: it records whatever the live server actually serves and labels it as such. The real-recording branch of `captureCompletionProof` remains, guarded by `!isolated`, for any future non-isolated action path. Test: `isolatedProofNote` unit test asserts the honest wording, branch interpolation, and the `/clip` pointer.
2. **Navigation not origin-confined after startup.** `recordAttempt` only ran the initial `canReach` probe; the Playwright recording context had no request routing, so a scored click could navigate off-origin or pull unapproved subresources. Fix: reused the hardened screenshot path's approved-origin policy — exported `isAllowedScreenshotResource` from `project-screenshot.ts` and, in `recordAttempt`, added `page.route("**/*", …)` that aborts any request whose URL is not an approved project origin (same `allowedOrigins` set the screenshot path builds via `projectScreenshotOrigins`), plus a post-`goto` URL check and a per-step `page.url()` check that breaks the flow if navigation left the approved origin. `canReach` already validates redirect targets against the allowlist. Verified via the existing build + the screenshot-origin helpers' existing tests; the routing predicate is the exact one the screenshot path is tested against.
3. **Watch could overlap captures and grow memory unboundedly.** `setInterval` had no in-flight guard and every PNG was appended to `frames` (12-cap only applied at GIF build). Fix: added a `capturing` flag so a tick is skipped while a capture is still running (no overlapping browser launches), and replaced `frames.push` with `pushBoundedFrame`, which trims to `WATCH_MAX_FRAMES` (12) as frames arrive so at most 12 PNG buffers are ever retained. Test: `pushBoundedFrame` unit tests assert the buffer never exceeds the cap across many pushes, keeps the most-recent frames, honors a custom cap, and mutates in place.
4. **Default-on watch too noisy.** Made watch opt-in: the `/do` handler now defaults `watch` to `false` (`getBoolean("watch") ?? false`), and the command option description says "default off". README + product plan updated to match.

Docs: README `/do` bullet and product-plan bullets rewritten to describe opt-in watch, the isolated-worktree honest-proof behavior, and full-session origin confinement.

`npm test`: first run showed the known flaky `security.test.ts` "empty temporary home" child-process case (5.0s timeout under load); a clean rerun passed **133/133** (130 baseline + 3 new: two `pushBoundedFrame`, one `isolatedProofNote`).

## Review round 2 (maintainer kept changes requested)

Bernard confirmed the three round-1 fixes (isolated skip note, in-flight guard + 12-frame retention, opt-in watch) and left 5 acceptance blockers. Each one → fix + test:

1. **Full-session origin confinement at context scope.** Replaced the page-scoped route with `confineRecordingContext(context, allowedOrigins)` installed on the `BrowserContext` before any page exists: context-scoped request routing (covers a popup's first request), `serviceWorkers: "block"`, popup/extra pages closed on the `page` event, off-origin top-level navigations close the page, and `context.routeWebSocket` proxies only sockets passing `isAllowedWebSocketUrl` (loopback + approved origin, no credentials) while closing everything else. Writing the end-to-end test bernard asked for exposed a real leak: `route.continue()` lets the browser follow a server redirect off-origin without re-invoking route handlers. Fixed with a shared `approvedOriginRouteHandler` (`route.fetch({ maxRedirects: 0 })` + fulfill, so the browser re-issues redirects as fresh routed requests) used by both the recording context and the hardened screenshot path, which had the same latent hole. Tests: `project-video.e2e.test.ts` serves a real local page with an external `target=_blank` link, off-origin subresource, off-origin redirect, and WebSocket attempt against a second recording server — asserts the off-origin server sees zero requests and zero upgrade attempts through a full `recordProjectFlow`, plus a focused test asserting popups are closed, off-origin sockets report `closed`, and a registered service worker never activates.
2. **`/clip` restricted to demonstrably non-mutating interactions.** `performFlowStep` now only navigates document links (`a[href]` whose href is not empty/`#`/script-scheme via `isNavigationHref`), scrolls, and hovers. Buttons, `[role='button']`, and `input[type='submit']` were removed from the candidate set entirely. Tests: e2e page carries a mutating button and a submit form posting to `/mutate`; the flow explicitly asks to "click delete account" and "submit order now" and the server asserts zero mutations; unit tests cover `isNavigationHref`.
3. **Isolated UI detection classifies the right checkout.** `runProjectRequest` now returns the isolated worktree's changed files (from `recordTaskWorktreeEvidence`, which returns them), and the proof decision moved into a pure exported `planCompletionProof(taskText, { isolated, branch, changedFiles })` that checks isolation with isolated-workspace files before any source-based classification; `listChangedFiles(source)` runs only for non-isolated tasks. Test: neutral wording ("Fix the bug") + UI-only isolated changed files → the isolated skip note; neutral wording + non-UI isolated files → no note.
4. **No raw navigated URLs echoed.** `recordAttempt` passes `page.url()` through the screenshot path's `safeReportedUrl` (exported), so `meta.finalUrl` in metadata, `/clip` replies, and proof notes never carries credentials, query strings, or fragments. Tests: unit test on `safeReportedUrl`; the e2e flow deliberately navigates to `/detail?token=supersecret#section` and asserts the reported URL is exactly the bare path.
5. **Real resource limits.** `recordProjectFlow` has a global concurrency guard (1 recording at a time, mirroring the screenshot limiter) and a hard abort deadline (`CLIP_HARD_TIMEOUT_MS`, overridable for tests) that schedules a forced `browser.close()` covering goto/waits/steps/retries and bounds the ffmpeg transcode via `execFile`'s `timeout`; Chromium and ffmpeg now launch with `minimalChildEnvironment()`. Tests: e2e hanging-page test asserts a 5s budget cuts the session long before goto's own 20s timeout with a "time budget" outcome; saturation test asserts a concurrent second recording is refused with a "recording limit" reason and that the slot is released afterwards.

Automatic proof for isolated tasks remains an honest skip; watch/timelapse stays labeled as the source dev server.
